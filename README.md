## Project Overview

This suite tests the Restful-Booker REST API across three quality dimensions:

| Dimension | Coverage |
|---|---|
| Authentication | Token generation, invalid credentials, edge cases |
| Functional | CRUD operations — GET, POST, PUT, PATCH, DELETE |
| Contract | Schema validation, required fields, data types |

**API Type:** RESTful, token-based authentication  
**Base URL:** `https://restful-booker.herokuapp.com`  
**Total test cases:** 16  
**Test files:** 2

## Test Strategy

The testing approach follows a **layered model**:

1. **Authentication first** — every CRUD operation depends on a valid token. Pytest collects `test_auth.py` before `test_bookings.py` (path order), which aligns with validating login before CRUD-heavy tests.
2. **Fixture-based setup** — `conftest.py` handles token generation at session scope (one token per test run) and booking creation/teardown at function scope (clean state per test).
3. **Positive and negative coverage** — every endpoint is tested with valid inputs, invalid inputs, and missing authentication to verify both happy paths and error handling.
4. **Schema validation** — response structure is validated on every GET to ensure contract stability.

> Tests are **independent and idempotent** — each test creates and cleans up its own data via fixtures, avoiding state pollution between runs.

## How to Run

### Prerequisites

- Python 3.12+
- pip

### Install dependencies

```bash
python -m pip install -r requirements.txt
```

### Set up environment

Copy the template and edit if needed:

```bash
cp .env.example .env
```

On Windows (PowerShell): `Copy-Item .env.example .env`

Use **`BOOKER_USERNAME` / `BOOKER_PASSWORD`**, not `USERNAME`: on Windows, `USERNAME` is already set by the OS and would override `.env` when using `python-dotenv` (and would break auth against the API).

```
BASE_URL=https://restful-booker.herokuapp.com
BOOKER_USERNAME=admin
BOOKER_PASSWORD=password123
```

### Run all tests

```bash
python -m pytest
```

(`pytest -v` also works if the `pytest` launcher is on your `PATH`; `python -m pytest` is reliable on Windows.)

### Run specific module

```bash
python -m pytest tests/test_auth.py
python -m pytest tests/test_bookings.py
```

### Run by marker

```bash
python -m pytest -m auth
python -m pytest -m functional
```

### View HTML report

After running, open `reports/report.html` in your browser.

## CI/CD Pipeline

The pipeline runs automatically on:
- Every `push` to `main`
- Every `pull_request` targeting `main`
- Scheduled: Monday–Friday at 8:00 AM UTC

```
Push to main / PR / schedule
    │
    ▼
GitHub Actions (api-tests job)
    ├── Setup Python 3.12
    ├── pip install -r requirements.txt
    ├── python -m pytest → reports/report.html
    ├── Optional zip artifact (same as before)
    └── On main only (not PRs): upload Pages artifact

If Pages artifact uploaded
    │
    ▼
deploy-pages job → public URL (see below)
```

### Live HTML report (GitHub Pages)

After each successful generation of `report.html` on **`main`** (push or scheduled run), the workflow publishes the report to **GitHub Pages**. Open it in the browser with one click — no zip download.

**URL:** [https://jennyloz89.github.io/restfulbooker-pytest/](https://jennyloz89.github.io/restfulbooker-pytest/)

**One-time repository setup**

1. In the repo go to **Settings → Pages**.
2. Under **Build and deployment**, set **Source** to **GitHub Actions** (not “Deploy from a branch”).
3. The first run may ask you to **approve** the `github-pages` environment; approve it so deploy can proceed.

Pull requests still run all tests but **do not** update the public site, so the Pages URL always reflects the latest **`main`** run that produced a report.
